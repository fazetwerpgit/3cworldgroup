Layout Blueprint
The portal utilizes a dense, full-width fluid layout optimized for data-heavy operations. The shell consists of a persistent left-hand navigation sidebar (240px desktop, collapsing to 64px icon-only on tablet, and a hidden drawer on mobile) and a 56px top app bar for global context (search, user profile, environment switcher). The main content area uses a 12-column fluid grid with an 8pt rhythm (16px margins on mobile, 24px on tablet, 32px on desktop). Dashboard: A top row of compact KPI sparkline cards, followed by a 2/3 and 1/3 split for the Active Pipeline data table and Leaderboard list. Admin Pipeline & Sales: Full-page data tables with sticky headers, dense row padding (8px vertical), and inline editing. Detail views utilize right-side slide-out panels (Sheets) rather than modals to maintain context. Calls & Email Templates: A split-pane architecture with a master list on the left (30% width) and a detail/editor view on the right (70% width). Onboarding: A structured wizard layout with a left-side vertical progress stepper and a max-width (800px) content area for form legibility. Responsive behavior: On mobile, split-panes stack vertically, data tables collapse into card-based list views, and the sidebar moves to a hamburger menu.

Visual System
The visual language is restrained, professional, and utilitarian, avoiding generic 'AI-slop' aesthetics. Typography: Inter is used universally for its high legibility in dense data environments. Font weights are strictly limited to Regular (400) for body, Medium (500) for UI elements and table headers, and SemiBold (600) for section headings. No oversized hero text. Spacing: A strict 8pt grid system (4px, 8px, 16px, 24px, 32px). Components use tight padding (e.g., 12px 16px for cards) to maximize screen real estate. Radii: Subtle 6px (rounded-md) for interactive elements and 8px (rounded-lg) for structural cards; no pill shapes. Color Tokens: Surface/Background: #F8FAFC (App background) and #FFFFFF (Card surface). Primary Brand/Navigation: 3C Navy #0A1F44 used for the sidebar background, primary text (#0F172A), and active states. Accent/Action: 3C Green #8dc63f used sparingly for primary action buttons, success states, and positive KPI trends. Borders: #E2E8F0 for structural separation. Icons: Lucide React icons exclusively, standardized to 16px for inline UI and 20px for navigation, replacing all emojis.

Motion Plan
Motion is strictly purposeful, serving to guide user focus and provide state feedback without unnecessary flair. Durations are kept short to maintain a snappy, productive feel. Micro-interactions (Hover/Focus/Active): 150ms duration, ease-in-out. Applied to button background color changes, table row highlights, and navigation item active states. Layout Transitions (Side Panels/Sheets): 300ms duration, cubic-bezier(0.16, 1, 0.3, 1) (ease-out). Used for the detail view slide-ins for Sales and Admin Pipeline to feel mechanical and precise. Data State Changes (Sort/Filter/Delete): 200ms duration, fade-in and subtle slide-up (4px) for list reordering. Accessibility: All motion is wrapped in a prefers-reduced-motion media query. If reduced motion is preferred, durations are forced to 0ms, relying solely on opacity and color state changes.

Implementation Notes
Target Stack: Next.js 16 App Router, React 19, Tailwind CSS v4, shadcn/ui, Firebase. Architecture: Use Route Groups (e.g., (portal)) to share the main dashboard shell layout. Leverage React 19's useTransition for optimistic UI updates during Firebase writes (e.g., updating pipeline status) to make the app feel instantaneous. Tailwind v4: Define the custom color tokens (--color-3c-navy: #0A1F44; --color-3c-green: #8dc63f) in the root CSS. shadcn/ui Customization: Strip default heavy drop shadows from the Card and Sheet components; replace with a 1px solid border (#E2E8F0) and a very subtle shadow-sm for a flatter, more mature look. Use the Table component extensively, overriding cell padding to 'p-2' for higher data density. Firebase Integration: Use Firebase Admin SDK in Next.js Server Components for initial data fetching to reduce client-side waterfalls, passing serialized data to Client Components that initialize real-time onSnapshot listeners only where live updates are critical (e.g., Leaderboard, Active Calls).
